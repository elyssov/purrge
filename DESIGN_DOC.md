## 1. МИССИЯ ДВИЖКА

### 1.1. Что это
Prometheus Engine — собственный кросс-платформенный воксельный движок общего назначения с полной разрушаемостью среды, процедурной генерацией и поддержкой множественных режимов камеры.

### 1.2. Зачем
Существующие воксельные решения (MagicaVoxel, Teardown engine, Voxel Farm) либо закрыты, либо заточены под конкретный жанр, либо не лицензируются. Рынок открытого/лицензируемого воксельного движка с современной графикой — ПУСТ.

### 1.3. Стратегическая цель
Создать движок, пригодный для:
- Кавайных аркад (PURRGE)
- Тактических стратегий (ОРПП:Апокалипсис)
- Survival horror (будущие проекты)
- Любого жанра, требующего разрушаемой среды

Лицензировать под роялти. Бесплатно на старте, процент с продаж готового продукта. Модель Unreal Engine.

### 1.4. Прецедент
Движок разработан человеком и ИИ. Это первый коммерческий воксельный движок, созданный в такой конфигурации. Сам факт — маркетинговый актив.

### 2.3. ECS (Entity Component System)
Архитектура на базе ECS. Каждый объект мира — сущность (Entity) с набором компонентов:
- `VoxelBody` — воксельное тело (ссылка на octree)
- `Transform` — позиция, вращение, масштаб
- `Physics` — физические свойства (масса, хрупкость, коэффициент трения)
- `Destructible` — параметры разрушаемости
- `Skeleton` — скелет для анимации
- `AI` — поведенческое дерево
- `AudioEmitter` — источник звука
- `Interactable` — доступные взаимодействия

## 3. ВОКСЕЛЬНАЯ ПОДСИСТЕМА

### 3.1. Определение вокселя

```rust
struct Voxel {
    material: u8,      // Индекс материала (0 = пустота, 1-255 = материалы)
    color: [u8; 3],    // RGB цвет
    flags: u8,         // Битовые флаги: разрушен, жидкость, светится, прозрачный
}
```

Размер одного вокселя в памяти: 5 байт. При хранении в octree — только непустые воксели.

### 3.2. Размеры и масштаб

**Базовая единица:** 1 воксель = 1 см в игровом мире.

**Ограничение на объект (v1.0):** максимальный размер одного разрушаемого объекта — **50×50×50 вокселей** (50×50×50 см в игровом мире).

Обоснование: 50³ = 125,000 вокселей на объект. В octree (при типичном заполнении 30-40%) — ~40,000-50,000 непустых вокселей. ~250 КБ на объект. Это позволяет:
- Держать в памяти сотни объектов одновременно
- Разрушать в реальном времени без просадки FPS
- Рассчитывать физику обломков на CPU

**Размеры типичных объектов:**

| Объект | Размер (воксели) | Заметки |
|--------|------------------|---------|
| Кружка | 8×8×10 | Полая внутри |
| Книга | 15×20×3 | Плоская |
| Ваза | 12×12×25 | Полая, тонкие стенки |
| Стул | 40×40×45 | Ножки тонкие, сиденье плоское |
| Телевизор | 50×30×5 | На пределе по одной оси |
| Холодильник | 30×30×50 | На пределе по высоте |
| Кот | 25×15×30 | Без хвоста. Хвост — отдельный сегмент |
| Пёс (мелкий) | 15×10×20 | — |

**Объекты крупнее 50×50×50:** составные. Шкаф = два объекта (верх + низ). Диван = три (сиденье + спинка + подлокотники). Каждый сегмент разрушается независимо. Соединены физическими связями (joints) — при разрушении одного сегмента остальные могут отделиться.

### 3.3. Масштабирование (дорожная карта)

| Версия | Макс. объект | Разрешение мира | Целевое качество |
|--------|-------------|-----------------|------------------|
| v1.0 (PURRGE) | 50³ | 1 воксель = 1 см | Кавай/шарж, стилизация |
| v2.0 (ОРПП:Апокалипсис) | 100³ | 1 воксель = 0.5 см | Детализированная тактика |
| v3.0 (будущее) | 200³+ | 1 воксель = 0.25 см | Приближение к фотореализму, 4K |

При v3.0 (200³ = 8 млн вокселей на объект) потребуется:
- LOD-система (Level of Detail): дальние объекты рендерятся в низком разрешении
- Стриминг чанков: подгрузка/выгрузка из RAM по мере необходимости
- GPU-сторонние octree: хранение и обход дерева на видеокарте
- Nanite-подобный подход: автоматическая деградация детализации с расстоянием

Это закладывается в архитектуру с v1.0, но реализуется поэтапно.

### 3.4. Sparse Voxel Octree (SVO)

Структура хранения: дерево, каждый узел делит пространство на 8 октантов. Листья — воксели или пустота. Ветки с полностью пустыми потомками — обрезаются (sparse).

```
Пример: объект 50×50×50
- Полный массив: 125,000 × 5 байт = 625 КБ
- SVO (заполнение 35%): ~43,750 вокселей × 5 байт + узлы дерева ≈ 250-300 КБ
- Экономия: ~50%
```

Для полых объектов (ваза, кружка, холодильник) экономия достигает 70-80%.

Операции на SVO:
- **Чтение вокселя:** O(log N) — обход дерева от корня к листу
- **Удаление вокселя (разрушение):** O(log N) — пометить как пустой, обрезать ветку если все потомки пусты
- **Raycast (для рендера и коллизий):** O(log N) на луч — быстрый обход с ранним выходом
- **Массовое удаление (взрыв):** O(R³ × log N) где R — радиус. Для R=5 вокселей — ~125 операций, мгновенно

## 4. РЕНДЕР

### 4.1. Метод
GPU raymarching через SVO. Не классическая растеризация полигонов — прямой обход воксельного дерева на GPU. Каждый пиксель экрана — луч, который проходит через octree и находит первый непустой воксель.

Преимущества:
- Нет генерации мешей (экономия CPU)
- Идеальная точность (каждый воксель рендерится)
- Естественные тени (shadow rays через тот же octree)
- Масштабируется с разрешением экрана, а не с количеством полигонов

### 4.2. Pipeline

```
Frame:
  1. Обновить octree (изменённые чанки)
  2. Upload octree → GPU buffer
  3. Primary rays: для каждого пикселя — raymarching через SVO → color + normal + depth
  4. Shadow rays: от каждого hit-point к источникам света → тени
  5. Ambient Occlusion: короткие лучи в полусфере → мягкие тени в углах
  6. Post-processing: bloom, tone mapping, color grading
  7. UI overlay
```

### 4.3. Освещение

**v1.0 (PURRGE):**
- Один направленный свет (солнце через окно)
- До 8 точечных источников (лампы в комнатах)
- Простой ambient occlusion (SSAO или voxel-based AO)
- Мягкие тени через PCF или poison disk sampling

**v2.0+ (ОРПП:Апокалипсис):**
- Глобальное освещение (voxel cone tracing)
- Динамические тени от всех источников
- Volumetric lighting (лучи через пыль и туман)
- Emissive materials (огонь, светящиеся руны, эктоплазма)

### 4.4. Система камер

Движок поддерживает несколько режимов камеры — переключение без перезагрузки сцены:

| Режим | Описание | Использование |
|-------|----------|---------------|
| ThirdPerson | За спиной/хвостом персонажа | PURRGE (основной) |
| SideScroller | Вид сбоку, фиксированная ось | PURRGE (уровень «Улица») |
| Isometric | Классическая изометрия | ОРПП:Апокалипсис (тактика) |
| TopDown | Вид сверху | ОРПП:Апокалипсис (геоскейп) |
| FirstPerson | От первого лица | Будущие проекты |
| Cinematic | Скриптованная камера | Кат-сцены |

Для PURRGE: основной режим — ThirdPerson, камера «над хвостом» кота. При переходе на уровень «Улица» — плавная трансформация в SideScroller.

### 4.6. LOD (закладка на будущее)

В v1.0 при масштабе объектов 50³ LOD не критичен — всё помещается в одну комнату. Но система LOD закладывается в архитектуру:

- LOD 0 (близко): полное разрешение, каждый воксель
- LOD 1 (средне): 2×2×2 воксели сливаются в один, цвет усредняется
- LOD 2 (далеко): 4×4×4
- LOD 3 (горизонт): 8×8×8

Переходы между LOD — плавные (blending). Для ОРПП:Апокалипсис и городских сцен — критически важно.

## 5. ФИЗИКА

### 5.1. Коллизии
Воксельные коллизии — проверка занятости воксельной ячейки. Быстрее полигональных в разы, так как octree даёт O(log N) доступ.

Типы коллизий:
- **AABB vs Octree:** персонаж (bounding box) vs мир (octree). Для ходьбы, прыжков
- **Ray vs Octree:** для стрельбы, взгляда, обнаружения. Raycast через SVO
- **Sphere vs Octree:** для взрывов, областей поражения
- **Object vs Object:** AABB vs AABB для столкновений объектов друг с другом

### 5.2. Разрушаемость

Каждый объект с компонентом `Destructible`:

```rust
struct Destructible {
    max_hp: f32,              // прочность объекта в целом
    current_hp: f32,
    voxel_hp_multiplier: f32, // прочность каждого вокселя = material.hardness × multiplier
    on_destroy: DestroyEvent, // что происходит при полном разрушении
    fragments_min: u8,        // минимум обломков
    fragments_max: u8,        // максимум обломков
    can_shatter: bool,        // разлетается на куски (стекло) vs ломается на части (дерево)
}
```

**Алгоритм разрушения:**
1. Удар/столкновение → рассчитать силу импакта (масса × скорость)
2. Если сила > порог материала → удалить воксели в зоне удара
3. Проверить связность: если часть объекта отделилась от основной массы — создать новый объект (обломок)
4. Обломок получает физику: гравитация, вращение, отскок
5. При ударе обломка о пол/стену — звук материала, частицы
6. Обломки < 3 вокселей — превращаются в частицы (экономия памяти)

**Проверка связности:** flood fill из центра масс. Все воксели, до которых нельзя дойти через соседей — отделяются. Это O(N) где N — количество вокселей в объекте. Для 50³ — мгновенно.

### 5.3. Цепные реакции

Системный принцип: каждый разрушенный/упавший объект проверяет, задел ли он другие объекты. Если задел — передаёт импульс. Те в свою очередь — могут разрушиться, упасть, задеть следующие.

```
Шар (падает с антресоли)
  → удар по вазе (стоит на холодильнике)
    → ваза разрушается (осколки + звон)
      → осколки задевают магниты на холодильнике (падают)
    → ваза задевает дверцу холодильника (открывается)
      → продукты падают (каждый — отдельный объект)
```

Максимальная глубина цепочки: не ограничена движком. Ограничивается только физикой (энергия затухает с каждым столкновением).

### 5.4. Жидкости (v1.0 — упрощённая модель)

Вода — не воксельная симуляция в реальном времени (слишком дорого). Упрощённая модель:
- Источник (кран) генерирует «поток» — анимированный набор вокселей
- При столкновении с полом — создаётся «лужа» (плоский слой вокселей, расширяется со временем)
- Лужа влияет на трение (кот скользит)
- Лужа повреждает электронику (ноутбук в луже = $1,200 ущерба)

## 6. АНИМАЦИЯ

### 6.1. Скелетная система

Персонажи (кот, пёс, хозяйка) имеют скелет — иерархию костей. Каждая кость управляет областью вокселей. При движении кости — воксели перемещаются и вращаются вместе с ней.

**Скелет кота (26 костей):**
```
Root
├── Spine1 (поясница)
│   ├── Spine2 (грудь)
│   │   ├── Neck
│   │   │   └── Head
```

| Анимация | Кости | Заметки |
|----------|-------|---------|
| Idle (стоит) | Всё тело + хвост виляет | Лёгкое покачивание |

| Fall death | Все кости ragdoll | Ангельское колечко |

### 6.3. Процедурные анимации
Некоторые анимации не записаны, а генерируются:
- **Хвост:** физическая симуляция (цепочка из 4 костей, реагирует на движение, гравитацию, «настроение»)
- **Уши:** поворачиваются к источнику звука (IK — inverse kinematics)
- **Глаза:** следят за ближайшим интересным объектом (рыбки, птица за окном, пёс)
- **Дыхание:** лёгкое расширение грудной клетки (синусоида)

### 6.4. Деформация воксельного тела (закладка на ОРПП:Апокалипсис)
Для монстров: берём человеческий скелет, программно деформируем (удлиняем кости, добавляем дополнительные, удаляем). Воксельное тело перестраивается вокруг нового скелета. В PURRGE не используется — но архитектура готова.

## 7. AI-ПОДСИСТЕМА

### 7.1. Навигация
A* по воксельной сетке. Навигационный граф строится из проходимых вокселей (пол, ступени). Обновляется при разрушении среды — новые проходы, заблокированные пути.

### 7.2. Восприятие (Perception)

```rust
struct Perception {
    hearing_radius: f32,       // радиус слуха (в вокселях)
    hearing_threshold: f32,    // минимальная громкость звука для реакции
    vision_cone: f32,          // угол обзора (градусы)
    vision_range: f32,         // дальность зрения
    memory_duration: f32,      // как долго помнит последнее обнаружение (секунды)
}
```

Для пса: слух — основной канал. Громкое разрушение → пёс просыпается → бежит к источнику звука. Зрение — вторичное (пёс видит кота и начинает патрулировать).

### 7.3. Поведенческие деревья (Behavior Trees)

**Пёс:**
```
Root (Selector)
├── [Condition: слышит громкий звук?]
│   └── Sequence: проснуться → потянуться → бежать к звуку → патрулировать 30 сек → вернуться
├── [Condition: видит кота у ценного предмета?]
│   └── Sequence: тявкнуть → подбежать → встать между котом и предметом
├── [Condition: кот рядом и спокоен?]
│   └── Sequence: подойти → лечь рядом → режим «бро»
└── [Default]
    └── Sequence: спать → видеть сны (пузыри)
```

## 9. КОНЦЕПЦИЯ

### 9.1. Название

### 9.2. Одна фраза

## 10. РЕЖИМ «КВАРТИРА» (3D)

### 10.1. Камера
Вид от третьего лица, «с хвоста». Камера за спиной кота, чуть выше, с лёгким отставанием. Вращение мышью/стиком. Тот же режим камеры, что будет в ОРПП:Апокалипсис для вида «из-за плеча бойца».

### 10.2. Редактор кота
При старте — простой редактор:
- Цвет шкурки (рыжий, чёрный, белый, серый, полосатый, трёхцветный, сиамский)
- Форма ушей (торчком, висячие, скоттиш-фолд)
- Длина шерсти (короткая, длинная, лысый сфинкс)
- Цвет глаз (зелёный, жёлтый, голубой, разноцветные)
- Хвост (длинный пушистый, короткий, бубликом)
- Размер (мелкий/средний/крупный — влияет на геймплей: мелкий быстрее но слабее, крупный медленнее но толкает тяжёлое)

Все параметры — модификация базовой воксельной модели (перекраска, масштабирование сегментов, замена формы ушей/хвоста из набора префабов).

### 10.3. Управление

**Движение:**
- WASD / левый стик — ходьба/бег
- Shift / зажим кнопки — красться (тихий режим)
- Space / A — прыжок
- Мышь / правый стик — камера

**Действия:**
- **[E / X] одинарное** — зацепить когтем / отпустить. Тянуть скатерть, цеплять занавеску, подтягивать предмет к краю
- **[E / X] двойное** — удар лапой с когтями. Ободрать, порвать, хлестнуть. Гонять пса (резко растёт охуевание — царапины)
- **[F / Y] одинарное** — толкнуть лапкой вперёд. Направление = куда смотрит кот. Основной инструмент для сбрасывания
- **[R / B] одинарное** — поссать. В любом месте
- **[R / B] двойное** — посрать. В любом месте. На подушке хозяйки — ядерный вариант
- **[Q / LB]** — мяукнуть (привлечь внимание пса)

### 10.4. Таймер
9 игровых часов = ~30-45 минут реального времени. Настенные часы в квартире — воксельные, стрелки двигаются.

## 11. ПРОЦЕДУРНАЯ ГЕНЕРАЦИЯ КВАРТИРЫ

### 11.1. Принцип
Модульная генерация: комнаты (прямоугольные тайлы) + коридоры + двери. Каждая комната имеет тип и обязательный набор предметов.

### 11.2. Типы комнат

**Прихожая (всегда):** дверь (заперта), вешалка, обувная полка, зеркало, коврик. Опционально: зонт, ключница.

**Кухня (всегда):** плита (у стены), холодильник (у стены), микроволновка, раковина, стол, посуда, ножи на магнитной планке. Опционально: кофеварка, тостер, блендер, ваза с фруктами.

**Ванная (всегда):** ванна или душевая (рандом), унитаз, раковина, зеркало, полотенца, рулон туалетной бумаги (ОБЯЗАТЕЛЬНО). Опционально: стиральная машина, корзина, весы.

**Спальня (в квартирах > студии):** кровать (у стены), шкаф, тумбочка, будильник. Опционально: ноутбук, книги, фоторамки.

**Гостиная (в квартирах ≥ двушки):** диван, телевизор (напротив дивана), журнальный столик, полка. Опционально: аквариум, пианино, картины, ковёр.

**Балкон (50% шанс):** стул, цветочные горшки, бельевая верёвка, перила. ВИД НА ГОРОД. Тизер AEGIS:Inferno при сидении на перилах.

### 11.3. Правило цепочек
Генератор ОБЯЗАН создать минимум одну цепную реакцию на комнату. Алгоритм: поставить ценный тяжёлый предмет → убедиться что рядом или выше есть неустойчивый объект, достижимый для кота. Если нет — добавить (полку, антресоль, стопку).

## 12. ДОСЬЕ ХОЗЯЙКИ

Генерируется с квартирой. 10 параметров влияют на ценность предметов: любит сериалы (ТВ ×3), художница (картины ×3), книгоманка (книги ×2), геймерша (консоль ×3), фитоняшка (коврик йоги ×2), кошатница (кошачьи предметы ×0), минималистка (всё ×2, мало предметов), барахольщица (всё ×0.5, много предметов), ностальгичка (фото ×5, стоимость $0 но охуевание MAX), растениеводка (цветы ×3).

Отображение: «Записная книжка кота» — каракули лапой, иконки с сердечками и крестиками.

## 13. ПРЕДМЕТЫ И СТОИМОСТЬ

Каждый предмет: название, стоимость ($), ценность для хозяйки (из досье), очки скуки, вес (лёгкий/средний/тяжёлый/неподвижный), хрупкость, шум при разрушении, тип материала.

Зелёный значок $ вылетает с числом при разрушении. Цепные реакции — бонус ×1.5 за каждый предмет в цепочке.

Финальный экран: детализированный чек на ремонт. Каждая позиция. Общая сумма. Рекорд. Кнопка «Поделиться».

---

## 14. СКУКА, ОХУЕВАНИЕ, ДЕВЯТЬ ЖИЗНЕЙ

**Скука:** 100→0 со временем. Разрушение восполняет. На нуле — game over (смерть от скуки).

**Охуевание:** 0→100 от разрушений. Зоны: зелёная (<25), жёлтая (25-50), оранжевая (50-75), красная (75-100). На 100 — выброшен → платформер.

**Девять жизней:** буквально. Падение с высоты, электричество, падение тяжёлого на голову. Теряет жизнь, возрождается. На нуле — game over.

## 15. ПЁС

Маленький, кавайный, трясётся. Спит 70% времени (воксельные пузыри снов). Просыпается на громкие звуки — 3 секунды на пробуждение (форы для кота). Патрулирует 30-60 секунд. Мешает крушить (встаёт между котом и предметом). Может случайно помочь (сам сбивает что-то в панике). Может дружить (режим «бро» если кот тихий). Эволюция по циклам: крепче спит → бдительнее → два пса → два пса + попугай.

## 17. КОНЦОВКИ И РОГАЛИКОВАЯ ПЕТЛЯ

**Улица (2D-платформер):** камера сбоку, улучшенный NES-стиль, те же воксели. Бежать, жрать голубей, уворачиваться от собак и дворника, найти новую хозяйку (контрастную с предыдущей). Новая квартира — новый цикл.

**Нарастание:** каждый цикл — сложнее. Квартира больше, пёс бдительнее, новые помехи.
